# Architecture Deep-Dive: Linux Enterprise Baseline Auditor

## 1. Project Objective
The goal of this project was to build a lightweight, automated endpoint auditing tool that mimics the initial data-collection phases of enterprise security agents (like CrowdStrike or Datadog). 

Rather than relying on third-party software, this tool leverages native Linux binaries and Python to extract system configurations, compare them against security baselines, and output structured threat intelligence.

## 2. System Architecture
This tool operates in a two-phase architecture to separate privilege requirements from logic processing:

* **Phase 1: The Collection Engine (Bash)**
  * Requires `root` execution to access restricted system logs and network states.
  * Focuses purely on raw data extraction without processing overhead.
* **Phase 2: The Logic Engine (Python)**
  * Runs with standard user privileges, adhering to the principle of least privilege.
  * Ingests the raw data, applies security logic, and formats the output.

## 3. Technical Breakdown

### Phase 1: Native Linux Data Extraction
The `collector.sh` script utilizes standard Linux administration tools to pull three critical data points:
* **Network State (`ss -tulpn`):** Maps all active TCP/UDP listening ports and the processes bound to them. This is crucial for identifying unauthorized web servers or legacy backdoors.
* **Authentication Logs (`grep` on `/var/log/auth.log` or `secure`):** Scrapes the system authorization logs specifically for "Failed password" strings. This detects active SSH brute-force attacks.
* **Service Configurations (`systemctl list-units`):** Dumps a list of all actively running `systemd` services to verify that mandatory security controls (like `ufw` or `firewalld`) are operational.

### Phase 2: Programmatic Security Analysis
The `auditor.py` script acts as the brain of the operation. It reads the raw text files generated by the Bash script and runs them through specific security checks:
* **Port Auditing:** Parses the network data to flag highly insecure legacy protocols, specifically looking for exposed Telnet (Port 23) and FTP (Port 21).
* **Threshold Alerting:** Counts the volume of SSH failures. A low number is categorized as background internet noise, while high volumes are flagged as active brute-force attempts.
* **Compliance Checking:** Verifies the presence of host-based firewalls, enforcing baseline enterprise compliance.

### Phase 3: Structured Data Export
To simulate enterprise environments where data must be ingested by a Security Information and Event Management (SIEM) system (e.g., Splunk, Elastic), the Python script utilizes the `json` and `datetime` libraries. 

It packages the analysis into a structured JSON payload, time-stamping the audit and categorizing findings logically, making the data instantly readable by automated alerting dashboards.

## 4. Real-World Application
This project demonstrates a practical understanding of:
1. **Linux System Administration:** Navigating processes, network sockets, and systemd services.
2. **Threat Detection:** Identifying Indicators of Compromise (IoCs) like anomalous SSH traffic and unauthorized open ports.
3. **Security Automation:** Translating manual terminal commands into an automated, repeatable pipeline.
